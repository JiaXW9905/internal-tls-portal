# 产业互联网内部门户 - LLM 快速上下文提示词（中文）

## 1. 使用说明
本文件用于在切换模型或新开会话时，快速让 LLM 获得必要上下文。  
建议做法：
- 先复制“基础上下文 Prompt”
- 再复制一个“任务模板 Prompt”
- 最后补充你的具体目标与限制

---

## 2. 基础上下文 Prompt（通用）

```text
你是该项目的协作工程师，请基于以下上下文工作：

[项目名称]
Internal TLS Certificate Portal

[技术栈]
- Node.js + Express（单体应用）
- SQLite（data/app.db）
- 前端为 public/*.html + public/*.js（原生 JS）
- Docker + docker-compose 部署

[关键目录]
- 后端入口：src/server.js
- 数据初始化：src/db.js
- 部署脚本：deploy/package.sh, deploy/docker-entrypoint.sh
- 编排配置：docker-compose.yml
- 持久化目录：data/, uploads/
- 文档目录：docs/zh-CN, docs/en

[当前核心业务规则]
1) 角色：admin/dev/service/product
2) 数据可见性：在申请/签发相关数据中，admin/dev 默认仅可见与自己相关的数据（自己提交或自己签发）
3) 证书撤销：签发超过7天不可撤销
4) 总览过期时间：无显式过期时间时，按 cert_created_at + 730天（365*2）计算
5) 文件缺失时，下载动作应不可操作并有明确提示
6) 状态更新逻辑：同 VID 发起新申请后，旧证书置为“已更新”（默认不展示、邮件不通知）
7) 页面右下角显示版本角标（public/version.js）

[质量要求]
- 不要随意改动既有业务规则
- 修改前先说明风险与影响面
- 修改后给出最小验证步骤（至少3条）
- 输出应包含：变更点、原因、验证方法、回滚建议
- 发版时应根据版本号打出安装包，修改各类文档
- 发版前检查因调试等原因增加的临时数据与功能，在发版中彻底剔除
- 每次与我对话时，说明当前所处分支名
```

## 3. 任务模板 Prompt（按场景）

### 3.1 功能开发 / 小需求
```text
请在不破坏现有业务规则的前提下实现以下需求：
管理员可以对‘申请人’为空的证书，进行一键删除/撤销

约束：
- 保持现有角色权限逻辑不变，除非我明确要求
- 每次与我对话时，说明当前所处分支名
- 保持 API 向后兼容，除非我明确同意
- 先给“改动方案摘要”，待我确认后再执行修改
- 修改后输出：
  1) 修改文件清单
  2) 风险点
  3) 验证步骤
```

### 3.2 Bug 修复
```text
请按“定位 -> 原因 -> 修复 -> 验证 -> 回滚”流程处理该问题：
1、我提交了2条名为’测试0302‘的证书删除申请，但在研发操作界面，该证显示的还是’上传并签发‘，你是否搞错了申请类型；
2、在研发的待处理申请中，需要针对每条申请展示’申请类型‘为新版，更新/续期，删除/注销

要求：
- 先定位根因，不要盲改
- 优先最小改动修复
- 明确说明是否影响历史功能
- 给出可执行的验证命令/步骤
- 在非main分支进行修改时，完成后直接重启localhost服务
- review所有可能导致这类问题的点，并进行彻底修复
- 每次与我对话时，说明当前所处分支名
```

### 3.3 发布前回归检查
```text
请基于该项目做发布前回归检查清单，并按高风险优先排序。

必须覆盖：
- 登录/注册/找回密码
- 申请 -> 签发 -> 下载主链路
- 证书总览筛选/分页/导出
- 7天撤销限制
- 数据可见性（admin/dev/service/product）
- 版本角标

输出格式：
- 用例名称
- 前置条件
- 操作步骤
- 预期结果
- 风险等级
```

### 3.4 升级与回滚方案审查
```text
我准备从 [旧版本] 升级到 [新版本]，请审查升级与回滚方案。

重点检查：
- 数据目录与上传目录的持久化路径是否正确
- env.prod 是否复用正确
- 端口与容器名冲突风险
- 回滚步骤是否可在10分钟内执行

输出：
1) 升级前检查项
2) 执行步骤（最短路径）
3) 失败回滚步骤
4) 发布后5分钟验证清单
```

## 4. 输出风格约束 Prompt（建议附加）

```text
请用中文输出，风格简洁、工程化，不要空泛描述。
请严格按以下结构输出：
1) 结论（1-3句）
2) 关键改动点（列表）
3) 风险与兼容性影响
4) 验证步骤（可执行）
5) 回滚建议
```

## 5. 一键组合示例（可直接粘贴）

```text
你是该项目的协作工程师，请基于以下上下文工作：
- 项目：产业互联网内部门户 v1.0.0
- 技术：Node.js + Express + SQLite + RBAC + Docker
- 架构：门户 + 多服务 + 细粒度权限
- 关键规则：
  1) TLS服务：tls-admin/tls-dev 默认仅可见自己相关数据
  2) TLS服务：签发超过7天不可撤销
  3) TLS服务：无过期时间时按 cert_created_at + 730天计算
  4) TLS服务：文件缺失时下载不可操作
  5) RBAC：权限命名格式 {service}:{resource}:{action}
  6) RBAC：所有权限变更需审计
  7) 门户：分层级可展开菜单

现在任务：[填写具体任务]

要求：
- 先给定位结论和最小改动方案，再给代码修改建议
- 输出必须包含：变更点、风险、验证步骤、回滚建议
- 全程中文，简洁明确
- 涉及权限变更时，说明RBAC表影响
```
